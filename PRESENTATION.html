Solution Overview

How It Works

A rule-based AI advisor that:

  • Analyzes monthly expenses against benchmarks
  • Assigns severity levels (P1-Critical to P5-Healthy)
  • Provides actionable, ranked savings recommendations
  • Answers natural-language questions via rule-based chatbot
  • Augments responses with relevant knowledge base articles

Agent Architecture

Multi-Agent Orchestration Pattern

User Input (Streamlit) ↓ FinanceAdvisorOrchestrator (Router) ├─► BackendAgent (analyze + chat) ├─► TriageAgent (severity + report) └─► FrontendAgent (charts) ↓ Output (Dashboard + Recommendations)

Key Innovation: Isolation Context

Each sub-agent receives only the minimum data it needs:

  • BackendAgent: sanitized customer snapshot only
  • TriageAgent: analysis output from BackendAgent
  • FrontendAgent: prepared data (pure functions)

Benefits

  • Security: Sandboxed execution, no data leakage
  • Testability: Independent unit testing per agent
  • Scalability: Agents can be deployed separately
  • Maintainability: Clear boundaries, no shared globals

Sub-Agents: Deep Dive

1. Backend Agent

Responsibilities: Data analysis, chatbot responses, RAG augmentation

  • Compute expense summary (income - expenses = savings)
  • Identify overspent categories vs benchmarks
  • Generate saving suggestions (top 3)
  • Rule-based chatbot with intent detection

2. Triage Agent

Responsibilities: Severity classification, report generation, QA gate

  • Classify overspend: P1-Critical (>60%) → P5-Healthy (within budget)
  • Generate markdown report with recommendations
  • Act as quality gate before UI rendering

3. Frontend Agent

Responsibilities: Visual payload preparation

  • Convert data → Plotly chart objects (Donut, Bar, Gauge, Sparkline)
  • Prepare responsive layouts for Streamlit
  • Pure functions (no side effects)

Context Management & Memory

Sub-Agent Memory Trimming (Step 8)

Challenge: Keep conversation history within context budget (~4,000 chars)

Solution: Sliding Window + Compression

Component Size Purpose
Session Summary ≤600 chars (15%) Compressed insights from all previous turns
Financial Snapshot ~200 chars Current income, savings, spending metrics
Recent Turns ~3,200 chars (80%) Latest 8–10 conversation turns (verbatim)

Benefits

  • Efficiency: No memory explosion on long conversations
  • Context Retention: Recent history + compressed summary
  • Isolation: Per-session instance (no data leaks between sessions)

Questions & Answers

Q: Why rule-based instead of LLM?

A: Fast (287ms vs 5–10s), explainable, privacy-preserving, cost-effective (₹2–5 vs ₹50–100 per user).

Q: How many concurrent users can the system handle?

A: Sustainably 20 VUs; burst to 50 with minor degradation; scales to 100+ with horizontal scaling.

Q: Is customer data encrypted?

A: Yes, at rest (planned: encrypted DB) and in transit (HTTPS). All data stays on-premise.

Q: What if the knowledge base is unavailable?

A: Chatbot still responds via rule-based engine; RAG enhancement just won't happen (graceful degradation).

Q: How is the system monitored?

A: OpenTelemetry exports traces/metrics/logs to Grafana; 5-panel dashboard tracks analyses, chats, latencies, and rates.

Q: Can this be customized for different customer segments?

A: Yes. Benchmarks and tips are configurable per segment; rule-based approach allows easy tuning.
